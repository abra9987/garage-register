## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (gap-1) | Icon-to-text gaps, inline badge margins |
| sm | 8px (gap-2 / p-2) | Compact form field gaps, field-row internal spacing |
| md | 16px (gap-4 / p-4) | Form field vertical gaps, card padding, PDF/form gutter |
| lg | 24px (gap-6 / p-6) | Page padding (matches existing main p-6), section separation |
| xl | 32px (gap-8 / p-8) | PDF preview internal padding |
| 2xl | 48px (py-12) | Review page header area |
| 3xl | 64px (py-16) | Not used this phase |

Exceptions:
- Touch targets on Approve/Unapprove buttons: minimum 48px height (h-12) per UIUX-03 "large buttons" requirement.
- PDF page navigation controls: 40px square (size-10) for comfortable click on desktop.

## Typography

| Role | Size | Weight | Line Height | Tailwind Class |
|------|------|--------|-------------|----------------|
| Body / Field Value | 14px | 400 (regular) | 1.5 | `text-sm font-normal leading-normal` |
| Label / Field Name | 14px | 400 (regular) | 1.5 | `text-sm font-normal text-muted-foreground` |
| Heading | 24px | 600 (semibold) | 1.2 | `text-2xl font-semibold` |
| Subheading | 16px | 600 (semibold) | 1.3 | `text-base font-semibold` |

Exactly 2 weights used: **400 (regular)** for body text, labels, field values, badge text; **600 (semibold)** for headings, subheadings, approve button text, section titles.

Notes:
- Continues Phase 2 typography contract exactly.
- VIN field uses `font-mono text-sm tracking-wider` for readability.
- Confidence badges use `text-xs font-normal` (12px, weight 400).
- Form input text uses `text-sm` (14px) matching shadcn Input default.
- Page heading uses `text-2xl font-semibold` matching upload page pattern.
- Original-value tooltip uses `text-xs font-normal` (12px).

## Focal Point

**Primary focal point: the form fields with confidence borders.** The entire review page exists so Andrey can scan field borders (green = skip, amber = glance, red = check against PDF, grey dashed = fill manually), correct errors, and click Approve. The side-by-side layout supports this by keeping the PDF visible while editing.

## Interaction Contracts

### Page Load Flow

1. User navigates to `/vehicles/[id]/review` (from upload page "Continue to Review" or from Register list)
2. Page fetches vehicle data + documents from API: `GET /api/vehicles/[id]`
3. While loading: skeleton placeholders in both PDF and form panels
4. On success: PDF renders in left panel, form populates with extracted values and confidence borders
5. On error (vehicle not found): empty state with "Go to Register" link
6. On error (network): error state with "Refresh" prompt

### Field Editing Flow (D-35)

1. User clicks on a form field -- standard focus behavior
2. User modifies value -- field border changes from confidence color to blue (`border-blue-500`)
3. "Edited" indicator appears below field in `text-xs text-blue-600`
4. Hovering over field shows Tooltip with "Original: {original value}"
5. Form dirty state activates -- "Save Changes" button appears above Approve
6. User clicks "Save Changes" -- PUT request to API, audit log entries created per changed field
7. Toast: "Changes saved"
8. Form resets dirty state, "Save Changes" button hides
9. Confidence borders remain blue for user-edited fields (they are now user-verified values)

### VIN Re-validation

1. User edits VIN field
2. On blur: client-side VIN validation runs (format + check digit)
3. VinStatus component updates below VIN input
4. If invalid: field does not prevent save (user may be entering partial data)

### Approval Flow (D-36, D-37, D-39)

1. User fills all required fields: VIN, Year, Make, Model, and at least one of Seller/Buyer Name
2. "Approve Record" button enables (removes `opacity-50`)
3. User clicks "Approve Record"
4. POST `/api/vehicles/[id]/approve` -- sets status to `approved`, creates audit log entry
5. Button shows spinner + "Approving..."
6. Toast: "Record approved -- ready for export"
7. Page re-renders: Approve button replaced by Unapprove button, status badge changes to "Approved"

### Unapproval Flow (D-38)

1. On approved record: "Return to Review" button visible instead of "Approve Record"
2. User clicks "Return to Review"
3. POST `/api/vehicles/[id]/unapprove` -- sets status back to `pending_review`, audit log entry
4. Toast: "Record returned to review"
5. Page re-renders: Approve button reappears, status badge changes to "Pending Review"

### Re-extraction Flow (D-32)

1. User clicks "Re-extract Document" in PDF panel
2. Toast: "Re-extraction will update AI-extracted values. Your manual edits are preserved."
3. Button shows spinner + "Re-extracting..."
4. POST `/api/vehicles/[id]/extract` -- re-sends PDF to Claude API
5. Poll for completion (same 2-second interval as Phase 2)
6. On success: form fields update with new extraction values. Fields previously edited by user retain user values (user edits take priority).
7. Toast: "Document re-extracted -- review updated fields"
8. On failure: Toast error, button returns to idle state

### PDF Navigation

1. PDF renders first page on load
2. Previous/Next buttons navigate pages
3. Page counter shows current position
4. Zoom In/Out buttons adjust scale (0.5x to 2.0x in 0.25 steps)
5. Document tabs (when both AP+AR exist) switch between documents
6. Switching document resets to page 1 of that document

## Accessibility

| Element | Requirement |
|---------|-------------|
| Form fields | All inputs have associated `<label>` via shadcn Form component. Required fields have `aria-required="true"`. |
| Confidence indicators | `aria-label="Confidence: {level}"` on each badge. Color is NOT the only indicator -- badge text label always present. |

| Status badge | `aria-label="Vehicle status: {status}"` |
| Action buttons | Clear visible text labels. No icon-only buttons for primary actions. |
| Toast notifications | Sonner handles `aria-live` automatically. |
| Edited field tooltip | Tooltip accessible via focus (not hover-only). `aria-describedby` linking input to tooltip content. |
| Form validation | Inline error messages with `aria-invalid="true"` and `aria-describedby` on invalid fields. |
| Keyboard navigation | Tab order follows visual field order. Enter submits form on last field. Escape closes tooltips. |
| Focus management | On page load, focus moves to first form field. After approve/save, focus returns to the action that was clicked. |
