**Implementation details:**

1. **Object URL lifecycle:** Use `useState` to store the blob URL. In a `useEffect` keyed on `[file]`:
   - Create URL via `URL.createObjectURL(file)`
   - Store in state
   - Return cleanup function that calls `URL.revokeObjectURL(url)`

2. **Container width measurement:** Use `useRef<HTMLDivElement>(null)` on the wrapper div. Use `ResizeObserver` in a `useEffect` (same pattern as review pdf-preview.tsx lines 60-72) to measure `contentRect.width` and store in state as `containerWidth`.

3. **Render structure:**
   ```
   <div ref={containerRef} className="bg-muted rounded-md overflow-hidden">
     {fileUrl && (
       <Document
         file={fileUrl}
         onLoadSuccess={handleLoadSuccess}
         onLoadError={handleLoadError}
         loading={<Skeleton className="h-[200px] w-full" />}
       >
         <Page
           pageNumber={1}
           width={containerWidth > 0 ? containerWidth : undefined}
           renderTextLayer={false}
           renderAnnotationLayer={false}
         />
       </Document>
     )}
   </div>
   ```

4. **onLoadSuccess handler:** Extract `numPages` from the callback argument `{ numPages: number }`. Call `props.onNumPages(numPages)`.

5. **onLoadError handler:** Call `props.onError()`. This lets the parent DropZone handle the error (show toast, revert to fallback).

6. **Loading state:** The `<Document>` `loading` prop renders `<Skeleton className="h-[200px] w-full" />` (import from `@/components/ui/skeleton`). Add `aria-busy="true"` to the wrapper div while `containerWidth === 0` or before `onLoadSuccess` fires. Track loading state with a `useState<boolean>(true)` that flips to `false` in `onLoadSuccess` and `onLoadError`.

7. **Accessibility:** Add `aria-label={`Preview of ${file.name}`}` on the wrapper div.

Export the component as a **named export**: `export function UploadPdfPreview(...)`.

**Dynamic import note:** This component uses `react-pdf` which requires browser APIs. The parent (DropZone) must use `next/dynamic` with `{ ssr: false }` to import it. Do NOT add dynamic import inside this file — that is the consumer's responsibility (Task 2).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client-side File API | User-selected files processed entirely in browser via URL.createObjectURL — no server involvement for preview |
| react-pdf renderer | PDF parsing happens in browser via pdf.js worker — untrusted PDF content rendered in sandboxed canvas |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Denial of Service | UploadPdfPreview | mitigate | Existing 10MB file size limit in react-dropzone config (maxSize: 10 * 1024 * 1024). PDF rendering is client-side only — large PDFs affect only the user's browser, not the server. Single page rendered (page 1 only) limits memory usage. |
| T-05-02 | Information Disclosure | URL.createObjectURL | mitigate | Object URLs revoked in useEffect cleanup (URL.revokeObjectURL) on file change and component unmount, preventing stale blob URL access. |
| T-05-03 | Tampering | PDF file content | accept | PDF is rendered by pdf.js in a canvas element (no DOM injection). pdf.js has its own security sandbox. File is user-selected from their own machine. No server-side rendering involved. |
| T-05-04 | Elevation of Privilege | react-pdf/pdf.js worker | accept | Worker runs in browser sandbox. No server-side code changes in this phase. Existing react-pdf dependency already in use (review page). No new attack surface. |
</threat_model>
